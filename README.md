# Example

Laravel 13 + Inertia v3 + Vue 3 application, styled with Tailwind CSS v4 and
shadcn-vue (`reka-sera` preset). Authentication is handled by Laravel Fortify;
typed routes are generated by Laravel Wayfinder.

## Requirements

- PHP 8.4
- Node.js (LTS) + npm
- [Laravel Herd](https://herd.laravel.com/) (serves the site at `http://example.test`)

## Getting started

```bash
composer setup   # install deps, copy .env, generate key, migrate, build assets
```

Then start the dev servers (Vite + queue + logs) with:

```bash
composer dev
```

The app is served by Herd at `http://example.test`.

## Development workflow

The golden rule: **run the checks before you commit, so CI is never red.**

### While you are working

Use the **auto-fix** commands — they rewrite files to match the project style:

```bash
npm run lint             # ESLint --fix (JS / Vue / TS)
npm run format           # Prettier --write (frontend)
vendor/bin/pint --dirty  # Laravel Pint (only the PHP files you changed)
```

Every change that adds or fixes behaviour must be covered by a test. Run just
the ones you touched for a fast loop:

```bash
php artisan test --compact --filter=SomeTest
```

### After you finish a feature / before you push

Run the **full gate** — one command that runs everything CI runs
(frontend lint + format + typecheck, then PHP style + PHPStan + the whole test
suite). If this is green, CI will be green:

```bash
composer ci:check      # or the npm alias: npm run check
```

> ⚠️ `ci:check` uses the **check-only** variants (they report and fail, they do
> not fix). So run the auto-fix commands above first, then `ci:check` to verify.

### What each check catches

| Command | Layer | Catches |
| --- | --- | --- |
| `npm run types:check` | Frontend | TypeScript / Vue type errors (e.g. a missing export). Vite/dev does **not** do this. |
| `composer types:check` | PHP | PHPStan / Larastan static analysis. |
| `npm run lint` / `format` | Frontend | ESLint + Prettier code style. |
| `vendor/bin/pint` | PHP | Laravel Pint code style. |
| `php artisan test` | PHP | Pest feature & unit tests. |

> **Note:** `types:check` means two different things: `npm run types:check` is
> the frontend (`vue-tsc`), `composer types:check` is the backend (PHPStan).
> `composer ci:check` runs both, so you rarely need to remember the difference.

## Code style & conventions

- **Strict types everywhere.** Every PHP file must start with
  `declare(strict_types=1);`. You do not add this by hand — Laravel Pint inserts
  it automatically via the `declare_strict_types` rule in `pint.json`. Just run
  `vendor/bin/pint` (or `composer lint`) and it is added for you. Do not remove it.
- **Architecture rules** are enforced by `tests/Unit/ArchTest.php` (Pest arch
  tests): controller / form-request / provider naming, concerns must be traits,
  no leftover `dd` / `dump` / `ray`, and the strict-types requirement above. When
  you introduce a new convention, add a rule there so it is enforced automatically.

## Working with shadcn-vue

`resources/js/lib/utils.ts`, `components.json`, the CSS theme variables in
`resources/css/app.css`, and everything under `resources/js/components/ui/` are
**managed by the shadcn-vue CLI** — applying or switching a preset can overwrite
them. Keep your own helpers in separate files (e.g. `resources/js/lib/url.ts`)
so the CLI never clobbers them. After running any `shadcn-vue apply` / `init`,
run `composer ci:check` to confirm nothing was silently overwritten.
